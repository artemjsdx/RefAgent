ОПИСАНИЕ ПРОЕКТА
────────────────────────────────────────────────────────────────────────────────
RefAgent — это автономный ИИ-агент, работающий через Telegram-бота.
Пользователь общается с агентом в чате, передаёт реферальную ссылку и условия
зачисления. Агент самостоятельно управляет пулом Telegram-аккаунтов
(через Telethon), выполняет цепочку действий (вступить в канал, запустить бота,
нажать кнопки), учится на ошибках через библиотеку знаний.

Среда разработки: Replit → деплой: Termux (Android)
Язык: Python 3.11+

БЛОК 2: КРИТИЧЕСКИЕ ПРАВИЛА (ВСЕГДА В СИСТЕМНОМ ПРОМПТЕ АГЕНТА)
================================================================================

⚠️  ПРАВИЛО №1 — API_ID / API_HASH
────────────────────────────────────
НИКОГДА не использовать один API_ID/API_HASH для нескольких аккаунтов.
Каждый .session файл должен подключаться со своим собственным API_ID и
API_HASH из sidecar .json файла рядом с сессией.
Если .json не найден → использовать значение из sessions.db для данного акка.
Если и там нет → пропустить аккаунт, не пытаться угадать.
Нарушение = массовая заморозка аккаунтов. Это необратимо.

⚠️  ПРАВИЛО №2 — ТАЙМИНГИ
─────────────────────────
- 60 сек между зачислениями рефов у одного реферера (жёсткий лимит)
- 15-30 сек между обработкой аккаунтов
- 5 сек после /start и /verify перед чтением ответа
- 10 сек максимальное ожидание ответа бота (5 попыток)

⚠️  ПРАВИЛО №3 — UID ФИЛЬТРАЦИЯ
───────────────────────────────
При загрузке сессий ПЕРВЫМ делом проверять UID.
Помечать аккаунты по категориям:
  - FRESH (UID > 8_500_000_000): ограниченные возможности
  - NORMAL (7_000_000_000 - 8_500_000_000): стандартные
  - OLD (< 7_000_000_000): максимальные права
  - CONDUCTOR: прогретый аккаунт, способный создавать группы

⚠️  ПРАВИЛО №4 — ПРОВОДНИК ОБЯЗАТЕЛЕН
───────────────────────────────────────
В тестах 0% аккаунтов смогли написать боту напрямую в DM.
Всегда использовать схему Harold-проводника:
  Проводник создаёт группу с ботом → фреш заходит по инвайту → общение в группе

⚠️  ПРАВИЛО №5 — БИБЛИОТЕКА ЗНАНИЙ
────────────────────────────────────
При любой ошибке СНАЧАЛА искать в /data/library/.
Если решение найдено → применять. Если нет → искать самому.
Если нашёл новое решение → записать в библиотеку.

⚠️  ПРАВИЛО №6 — ПЛАН ПЕРЕД РАБОТОЙ
─────────────────────────────────────
Никогда не начинать выполнение задачи без утверждённого плана.
Агент может обсуждать с пользователем, уточнять условия — но кнопка
"Запустить" недоступна пока план не создан и не показан пользователю.
Во время работы агент МОЖЕТ изменять план, уведомляя пользователя.

БЛОК 4: ДЕТАЛИ РЕАЛИЗАЦИИ
================================================================================

ПРОВОДНИК (Harold Pattern)
──────────────────────────
1. При загрузке сессий → автоматически тестировать:
   can_resolve_username() + can_create_group()
2. Лучший кандидат (старейший по UID, прошедший тест) = conductor
3. Кэш групп: {fresh_phone: {chat_id, bot_username, created_at}}
4. Cleanup: удалять группы после завершения задачи (экономия ресурсов)
5. Если conductor недоступен → агент уведомляет пользователя, не начинает работу

ОПРЕДЕЛЕНИЕ ТИПА БОТА
──────────────────────
1. Агент отправляет /help или /start в группу через conductor + fresh
2. Ждёт 7 сек
3. Если ответ есть → кэшировать: bot_@username = ACCEPTS_GROUPS
4. Если нет → DIRECT_DM_ONLY → уведомить пользователя об ограничении
5. Результат сохраняется в library/bot_@username.md

INLINE КНОПКИ
─────────────
- Всегда проверять reply_markup в ответе бота
- KeyboardButtonCallback → GetBotCallbackAnswerRequest
- URL-кнопки → логировать URL, не кликать автоматически
- ReplyKeyboard → SendMessageRequest с текстом кнопки

ОЖИДАНИЕ ОТВЕТА БОТА
─────────────────────
- Метод: send → asyncio.sleep(5) → iter_messages(limit=10)
- Максимум 5 попыток с интервалом 2 сек
- Если нет ответа за 10 сек → timeout error → в библиотеку знаний

PAGINATED MODEL BROWSER
───────────────────────
- GET https://openrouter.ai/api/v1/models → кэш на 1 час
- Разбить на страницы по 10
- Фильтры: Free (pricing.prompt == "0") / Paid
- Кнопки навигации: [◀ Назад] [1/30] [Вперёд ▶]
- Кнопка ввода вручную всегда доступна

FAVORITEAPI CONTEXT MANAGEMENT
────────────────────────────────
- При каждом ответе читать context_kb
- При >150/180KB → автоматически сжимать через write:ctx тег
- При CTX_LIMIT_180 → reset → restore → load:mem
- Все теги ([⊕...⊕]) скрыты от пользователя

TERMINAL TOOL
─────────────
- Агент может писать временные Python-скрипты в /scripts/
- Имя файла: tmp_{timestamp}_{description}.py
- Автоочистка скриптов старше 24 часов
- Таймаут выполнения: 60 сек по умолчанию, настраиваемо

sessions.db
───────────
CREATE TABLE accounts (
    id INTEGER PRIMARY KEY,
    phone TEXT UNIQUE,
    uid INTEGER,
    api_id INTEGER,          -- СВОЙ для каждого аккаунта!
    api_hash TEXT,           -- СВОЙ для каждого аккаунта!
    session_path TEXT,
    format TEXT,             -- TELETHON / TDESKTOP / PYROGRAM
    status TEXT,             -- active / frozen / banned / untested
    is_conductor INTEGER DEFAULT 0,
    warmup_score INTEGER DEFAULT 0,
    uid_category TEXT,       -- OLD / NORMAL / FRESH
    notes TEXT,
    added_at TIMESTAMP,
    last_used TIMESTAMP
);

results.db
──────────
CREATE TABLE tasks (
    id INTEGER PRIMARY KEY,
    ref_url TEXT,
    ref_code TEXT,
    conditions TEXT,
    plan TEXT,
    started_at TIMESTAMP,
    finished_at TIMESTAMP,
    total_accounts INTEGER,
    success_count INTEGER,
    fail_count INTEGER,
    status TEXT              -- running / completed / stopped / error
);

CREATE TABLE task_results (
    id INTEGER PRIMARY KEY,
    task_id INTEGER,
    account_id INTEGER,
    status TEXT,             -- success / failed / skipped
    fail_reason TEXT,
    timestamp TIMESTAMP,
    FOREIGN KEY (task_id) REFERENCES tasks(id),
    FOREIGN KEY (account_id) REFERENCES accounts(id)
);
